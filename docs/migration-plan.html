Phase 1：全量迁移（所有表）

使用 MySQL Shell 将 NA 库全量导出，传输到 EU 端后并行导入。覆盖所有三类表。

# NA 端导出
mysqlsh -- util dump-instance /backup/full_dump \
  --threads=8 \
  --compression=zstd \
  --excludeSchemas=["mysql","sys","information_schema","performance_schema"]

# EU 端导入
mysqlsh -- util load-dump /backup/full_dump \
  --threads=8 \
  --resetProgress \
  --ignoreExistingObjects=true

Phase 2：地域专属表数据清理

在第三类表中增加 data_region 字段，按业务规则标记后，各端删除不属于本区域的数据。

-- 1. 增加地域标识字段
ALTER TABLE talent ADD COLUMN data_region VARCHAR(4) DEFAULT 'NA';

-- 2. 按业务规则标记数据归属
UPDATE talent SET data_region = 'EU' WHERE <EU识别条件>;

-- 3. 各端删除对方数据
DELETE FROM talent WHERE data_region = 'NA';  -- EU 库执行
DELETE FROM talent WHERE data_region = 'EU';  -- NA 库执行

Phase 3：数据校验

分类验证迁移结果的正确性和完整性。

校验维度预期结果
独立配置表两端表结构一致
共享表checksum 对比，两端数据完全一致
专属表NA 记录数 + EU 记录数 = 迁移前总数

3. 技术选型：MySQL Shell vs mysqldump

维度MySQL Shellmysqldump
导出方式 多线程并行（8+ 线程） 单线程串行
压缩 原生 zstd（体积降 70%+） 需外部 gzip
断点续传 支持 不支持
导入速度 多线程 LOAD DATA 单线程逐条 INSERT
MySQL 8.0+ 原生支持 支持
结论：MySQL Shell 在大数据量场景下速度提升 5~10 倍，且支持断点续传，是生产级迁移的推荐工具。
